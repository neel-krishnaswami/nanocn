# Parse error messages via Menhir — summary

A summary of Menhir's "new way" for parse-error handling
(`menhir --manual` §11, §15), tailored to the shape of the nanoCN
parsers already in tree.

## What we have today

`lib/parser.mly` + `lib/smtParser.mly` are both compiled to the
table back-end (`MenhirLib.Convert.Simplified.traditional2revised`
in `lib/parse.ml` / `lib/smtParse.ml`). On error the driver catches
`Parser.Error`/`SmtParser.Error` and prints a generic

    parse error at <file>:<line>:<col>

i.e. the position is accurate but the message is state-independent.

## What the new way gives us

A `.messages` file that maps every possible LR error-state to a
hand-written diagnostic. Menhir can:

- enumerate every error state as a shortest input sentence that
  reaches it,
- check on each rebuild that the grammar has not grown a new
  reachable error state without a message (CI gate),
- compile the file into an OCaml function `message : int -> string`
  that the driver calls with the state number at error time.

No runtime cost beyond a table lookup; the intelligence lives in
the `.messages` file.

## The three file formats

### 1. Sentence

A terminal sequence that leads the parser to a particular error
state, prefixed with the start symbol:

    program: LET IDENT COLON IDENT

### 2. Entry in a `.messages` file

Four parts — a header with the start symbol's name and the sentence,
the LR-state position comment (automatically regenerated), a
separating blank line, then one or more lines of the message:

    program: LET IDENT COLON IDENT

    ##
    ## Ends in an error in state: 42.
    ##
    ## let_decl -> LET . ident_var COLON sort EQUAL expr [ EOF FUN RFUN ... ]
    ##
    ## The known suffix of the stack is as follows:
    ## LET
    ##

    After `let`, I expect an identifier to be introduced, followed by
    `= <expr>`.

The `##` lines are regenerated by `--update-errors`; the prose below
the blank line is hand-written.

### 3. Compiled OCaml (auto-generated)

`menhir --compile-errors parser.messages parser.mly > parser_messages.ml`
produces:

    let message : int -> string = function
      | 42 -> "After `let`, I expect an identifier..."
      | ...
      | _ -> raise Not_found

## Workflow commands

| Phase | Command | What it does |
|---|---|---|
| Bootstrap | `menhir --list-errors parser.mly > parser.messages` | Enumerate every error state; emit a skeleton entry for each. Messages are all literal `"<YOUR SYNTAX ERROR MESSAGE HERE>"` placeholders. |
| Refresh after grammar change | `menhir --update-errors parser.messages parser.mly > parser.messages.new && mv …` | Regenerate `##`-comments (state numbers, known-suffix) while preserving hand-written messages. |
| Maintain coverage | `menhir --list-errors parser.mly > expected.messages; menhir --compare-errors expected.messages --compare-errors parser.messages parser.mly` | Check `parser.messages` covers every reachable error state; fail CI if not. Also checks that every entry in `parser.messages` is still reachable (no stale entries). |
| Compile | `menhir --compile-errors parser.messages parser.mly > parser_messages.ml` | Emit the OCaml lookup function. |
| Inspect | `menhir --echo-errors parser.messages` / `--echo-errors-concrete` | Print just the sentences (optionally with token-alias substitutions) for review. |

The four maintenance steps (`list-errors`, `update-errors`,
`compare-errors`, `compile-errors`) are the whole pipeline.

## Integrating with a running parser

Two APIs to choose between.

### Option A — the incremental API (recommended)

Menhir already generates an `Incremental` sub-module when the
`--table` back-end is used. The existing `traditional2revised`
convert is a thin wrapper over it; switching to the incremental API
lets us intercept the failing checkpoint and recover the state
number.

Outline (replacement for `parse_raw` in `lib/parse.ml`):

```ocaml
let parse_raw entry s ~file =
  let buf = Lexer.from_string s ~file in
  let supplier = supplier buf in
  let rec loop checkpoint =
    let open MenhirInterpreter in
    match checkpoint with
    | InputNeeded _ ->
      let tok = supplier () in
      loop (offer checkpoint tok)
    | Shifting _ | AboutToReduce _ ->
      loop (resume checkpoint)
    | HandlingError env ->
      let state = current_state_number env in
      let msg =
        try Parser_messages.message state
        with Not_found -> "unknown parse error"
      in
      let pos = Lexer.pos_of_lexbuf buf in
      Error (Format.asprintf "%a: %s" SourcePos.print pos msg)
    | Accepted v -> Ok v
    | Rejected -> Error "parser rejected"
  in
  let start_pos, _ = Sedlexing.lexing_positions buf in
  loop (entry start_pos)
```

Here `entry` is the *incremental* entry point (e.g.
`Parser.Incremental.prog_eof`) rather than the traditional one —
the module name and signature are mechanical transformations.

### Option B — `--exn-carries-state` (code back-end)

For parsers on the code back-end, `menhir --exn-carries-state`
makes `exception Error` carry the state number:

    exception Error of int

The driver can then catch `Error n` and look up the message
directly. This is simpler code but is only available for the code
back-end; since nanoCN uses the table back-end we'd pick Option A.

## Dune wiring

Two additions to `lib/dune`:

1. A custom rule that runs `menhir --compile-errors parser.messages
   parser.mly` and produces `parser_messages.ml`.
2. A check alias that runs `--compare-errors` against a fresh
   `--list-errors` output and fails if they diverge — so CI catches
   added error states without a matching message.

Roughly:

```
(rule
 (targets parser_messages.ml)
 (deps parser.messages parser.mly)
 (action
  (with-stdout-to %{targets}
    (run menhir --compile-errors %{dep:parser.messages} %{dep:parser.mly}))))

(rule
 (alias check-messages)
 (deps parser.messages parser.mly)
 (action
  (chdir %{project_root}
    (bash
      "menhir --list-errors parser.mly > $(mktemp /tmp/expected.XXX) && \
       menhir --compare-errors parser.messages \
              --compare-errors $(mktemp /tmp/expected.XXX) \
              parser.mly"))))
```

## Writing good messages

Menhir's manual recommends using the `##`-block to read off what
the parser has seen (the "known suffix of the stack") and what it
expected (the LR item with its lookahead set), and writing a
message that references both in the user's vocabulary. Concretely:

- Name the construct we were part-way through recognising. The
  known-suffix line tells us which nonterminal's production we're
  in and how far we've got.
- List a small set of reasonable next tokens in plain English.
  Don't list every terminal in the LR lookahead set — pick the
  ones a user would plausibly type.
- If the state is reached only via a particular construct (e.g. an
  `rfun` domain entry after `[res]`), say so — the message is
  allowed to be specific because each state is its own entry.
- Keep messages one or two sentences; let the position information
  supplied by the driver do the locating work.

A good rule of thumb: if the same message would be right for two
states, either the two states are genuinely ambiguous (rare) or the
two messages should differ by naming a different surrounding
construct.

## Proposed work breakdown for nanoCN

1. **`parser.messages` bootstrap**: run `menhir --list-errors lib/parser.mly` and commit the skeleton. Every entry gets the literal default message, which we normalise to include the string `PLACEHOLDER` so remaining work is `grep -l PLACEHOLDER`-discoverable and CI can optionally fail on any placeholder that escapes.
2. **Incremental-API driver**: replace `parse_raw` in `lib/parse.ml` with the Option A outline. Do the same for `lib/smtParse.ml`.
3. **Dune rules**: compile-errors and compare-errors aliases for both `parser.mly` and `smtParser.mly`, backed by `parser.messages` and (trivially small) `smtParser.messages`.
4. **Fill in messages**: go through `parser.messages` one state at a time, writing a sentence-or-two message per state. The first pass should aim to cover *all* states — if the placeholders outlive the first pass on more than a small fraction of states, the pass was too conservative. Leaving large swathes of the grammar with `PLACEHOLDER` defeats the point: users hit arbitrary states, not a curated "common path", and a generic placeholder is barely better than the pre-existing `parse error at <pos>` we're replacing. Treat a state that's genuinely hard to describe as a signal the grammar shape deserves a comment in `parser.mly`, not a licence to skip it.
5. **CI gate**: add the compare-errors alias to the default `dune runtest` so added grammar rules surface as failing builds until the author adds a message. Optionally add a second check that `grep -c PLACEHOLDER parser.messages` is zero (or below some soft target) so placeholders don't accumulate.

Step 1 is the only one that's mechanical and easy to commit in
isolation; steps 2–3 are a one-afternoon scaffold; step 4 is the
open-ended work that produces the actual user-facing value; step 5
keeps the messages from rotting.
