# Plan: editor support for nanoCN (tree-sitter + LSP + Emacs)

## 0. Goal and scope

Deliver three cooperating artifacts so `.cn` and `.rcn` files get a
first-class Emacs experience:

1. **Tree-sitter grammar** (`tree-sitter-nanocn`) — loadable shared
   object that Emacs's `treesit` module can use for syntax
   highlighting, indentation, Imenu, and sexp navigation.
2. **LSP server** (`nanocn-lsp`) — an OCaml executable built on the
   `lsp`/`jsonrpc` opam libraries that re-uses the existing
   typechecker, provides diagnostics, hover (including
   "context at point"), go-to-definition, and reports SMT results
   asynchronously as they arrive.
3. **Emacs mode** (`nanocn-ts-mode` + Eglot glue) — a tree-sitter
   major mode plus Eglot registration that wires (1) and (2)
   together.

No changes are proposed to `doc/syntax.ott`. Everything here is
tooling; the language is unchanged.

## 1. Architectural overview

```
       Emacs buffer (nanocn-ts-mode)
               |
               |  tree-sitter: highlight / indent / imenu / sexp
   +-----------+-----------+
   | parser.so (from        |
   | tree-sitter-nanocn)    |
   +------------------------+
               |
               |  Eglot: LSP messages over stdio
   +-----------+-----------+
   |  nanocn-lsp executable |
   |  -- uses `lsp` library |
   |  -- dispatches to:     |
   |     * lib/lspCompile   |  (per-decl elaborate + check)
   |     * lib/hoverIndex   |  (spatial tree index)
   |     * lib/smtAsync     |  (background Z3 + streaming)
   |  -- publishes diag's   |
   |     via publishDiag's  |
   +------------------------+
               |
               |  pipe
   +-----------+-----------+
   |   z3 (one subprocess   |
   |   per file save)       |
   +------------------------+
```

Key separations:

- **Tree-sitter handles purely syntactic concerns** (highlighting,
  indentation, structural navigation). It also gives Emacs a
  robust parse tree even when the file is broken, which the LSP
  server does *not* rely on — the server has its own parse path.
- **The LSP server owns all semantics.** It re-parses (with
  Menhir) on save/change and runs the typechecker. It never looks
  at tree-sitter output.
- **The SMT loop is asynchronous relative to the rest of the
  server.** When Z3 is busy, the server keeps answering hover and
  completion requests; diagnostics pushed by Z3 arrive in later
  `publishDiagnostics` messages.

## 2. Piece A — Tree-sitter grammar

### 2.1 Layout

A standalone sub-project at `scripts/editor/tree-sitter-nanocn/`:

```
tree-sitter-nanocn/
├── grammar.js           # the grammar DSL
├── package.json         # required by tree-sitter-cli
├── queries/
│   ├── highlights.scm   # node captures → font-lock faces
│   ├── indents.scm      # (optional; treesit uses its own DSL)
│   ├── locals.scm       # scope / reference resolution for Emacs
│   └── tags.scm         # Imenu / xref tags
├── test/corpus/         # golden syntax-tree expectations
│   ├── examples/        # copies of examples/*.{cn,rcn}
│   └── errors/          # intentional syntax errors
└── README.md
```

Build artifacts (`src/parser.c`, `src/grammar.json`,
`src/node-types.json`, any `*.so`) are generated by
`tree-sitter generate` / `tree-sitter build` and `.gitignored`
(except `parser.c`, which we will commit so Emacs users don't need
a JS toolchain to rebuild).

### 2.2 One unified grammar for both `.cn` and `.rcn`

`parser.mly` today has two start symbols, `prog_eof` (surface) and
`rprog_eof` (refined). They share `SortDecl`, `TypeDecl`, and the
expression grammar; they diverge on `FunDecl` vs `RFunDecl` and on
the refined-specific constructs (`let log`, `let res`, `let core`,
`exfalso`, `open-take`, proof spines, etc.).

**Decision:** one grammar, with a top-level `source_file` rule
that accepts both surface and refined forms. Rationale:

- Mechanical 1:1 translation of `parser.mly` (which already
  unions the two) is simpler than maintaining two 80%-overlapping
  grammars.
- Emacs users editing either file type get the same mode; the
  LSP backend is the authority on which forms are legal.
- Tree-sitter only supports one start symbol per grammar, so
  two start symbols would require two grammars anyway.

### 2.3 Translating `parser.mly` to `grammar.js`

- **Precedence**: the cascading `or_expr → and_expr → eq_expr →
  cmp_expr → add_expr → mul_expr → app_expr` chain maps directly
  onto `prec.left(N, ...)` on binary-op rules.
- **Keywords vs identifiers**: Emacs/tree-sitter expects a
  `word: $ => $.identifier` declaration so that keyword matching
  is done at the lexer layer. Our lexer already distinguishes
  lower-initial `IDENT` (keywords live here) from upper-initial
  `LABEL` (constructors).
- **Hyphenated keywords** (`open-ret`, `open-take`, `make-ret`,
  `make-take`): tree-sitter's default tokenizer doesn't treat `-`
  as part of a word, so these must be defined as explicit
  multi-character token rules: `open_ret: $ => token('open-ret')`.
- **Comments** (`// ...`): declared in `extras`.
- **Error recovery**: rely on tree-sitter's built-in recovery;
  no explicit recovery productions needed.
- **No external scanner needed**: layout-insensitive, no strings,
  no indentation rules. A pure DSL grammar suffices.

### 2.4 Queries

- `highlights.scm` — captures like `(fun_decl name: (identifier)
  @function)`, `(constructor) @type`, keywords `@keyword`,
  comments `@comment`.
- `tags.scm` — top-level defuns (`fun_decl`, `rfun_decl`,
  `sort_decl`, `type_decl`) for Imenu.
- `locals.scm` — minimal; real scope resolution belongs to the
  LSP server. Useful only for local variable highlighting that
  doesn't require semantic info.

### 2.5 CI gate against `parser.mly` drift

A dune rule `(alias check-grammar)` runs `tree-sitter parse` on
every `examples/*.{cn,rcn}` and fails if any file produces `ERROR`
or `MISSING` nodes. This catches most drifts: the first time
`parser.mly` learns a new construct and tree-sitter's grammar
doesn't, examples will start failing.

This rule is best-effort — it doesn't detect *lost* productions
that no example exercises. Add a second gate in
`doc/instructions/grammar-change.md` that reminds edits to
`parser.mly` to update `tree-sitter-nanocn/grammar.js`.

### 2.6 Module-level design

Pure grammar project in JavaScript — nothing to express in OCaml
`.mli` files. The `.scm` query files *are* the module boundaries;
keep them short and well-commented.

## 3. Piece B — LSP server

### 3.1 External libraries

The existing ecosystem provides the three things we'd otherwise
hand-roll:

- **`jsonrpc`** (opam) — implements JSON-RPC 2.0 framing.
- **`lsp`** (opam) — "LSP protocol implementation in OCaml,
  designed to be as portable as possible and does not make any
  assumptions about IO." Provides typed client/server
  request/notification ADTs (`Client_request`, `Server_request`,
  `Client_notification`, `Server_notification`), LSP types
  (`types.ml`, generated from the LSP spec), and `Text_document`
  (handles incremental `didChange`, UTF-8/UTF-16 encoding).
- **`lsp.Io.Make` functor** — parameterizes packet I/O over an
  IO monad. Lets us plug our synchronous stdio directly in
  without pulling Jane Street's Fiber.

**We do not use `lsp-fiber` or `jsonrpc-fiber`.** They would
introduce a concurrency framework we don't need. Our main loop is
a simple `Unix.select` on stdin plus worker-thread wakeup pipes.

### 3.2 Compiler-side refactors (the hardest work)

The typechecker today is whole-program: `Parse.parse_prog` →
`Typecheck.check_prog` → `Error` or `Ok` for the entire file.
LSP needs finer granularity.

#### 3.2.1 Per-decl parsing with error recovery

Menhir's error recovery is weak. We avoid it with a
depth-tracking splitter:

```
val split_decls :
  string -> file:string -> decl_span list
  (* where decl_span = {
       kind : [`Fun | `Rfun | `Sort | `Type | `Main];
       start : int; stop : int;  (* byte offsets *)
       text : string;
     } *)
```

The splitter scans the file once, tracking brace/paren/bracket
depth and comment state, and emits a span whenever a line at
depth 0 starts with one of `fun`, `rfun`, `sort`, `type`, `main`.
Each span is then parsed independently with the *existing*
`Parse.parse_decl` (for `.cn`) or a new `Parse.parse_rdecl` (for
`.rcn`) and typechecked with `Typecheck.check_spec_decl` (which
already exists for the REPL).

Refined side: `check_rprog` is monolithic today; we extract a
`RCheck.check_rdecl : RSig.t -> RProg.decl -> (RSig.t *
Constraint.typed_ct) ElabM.t` that does the per-decl work. The
whole-program `check_rprog` is rewritten as a fold over
`check_rdecl`.

#### 3.2.2 Compile-one-file

New top-level driver:

```
(* lib/lspCompile.mli *)
type decl_result = {
  span : Splitter.decl_span;
  diagnostics : Error.t list;       (* empty on success *)
  core : Typecheck.typed_ce Prog.core_decl option; (* None on error *)
  constraints : Constraint.typed_ct list;  (* refined only *)
}

type file_result = {
  final_sig : Typecheck.typed_ce Sig.t;
  final_rsig : RSig.t;
  decls : decl_result list;
}

val compile_file : string -> file:string -> file_result
```

A parse-failed span contributes a parse diagnostic and the
`Sig`/`RSig` accumulator is unchanged (so later decls see the
world as if the failed one weren't present). A type-failed span
contributes a type diagnostic and, where possible, adds a stub to
the `Sig` based on the decl header (function signature without
body) so downstream decls that call it still elaborate. Stub
heuristics are best-effort; in the worst case, dependent decls
fail with "unknown function" and that's acceptable.

This replaces neither the existing CLI nor the current `bin/main.ml`
code paths — `check_prog` continues to exist. `compile_file` is
a new consumer of the same primitives.

#### 3.2.3 Hover index

After a successful per-decl typecheck we have a `typed_ce` tree
where every node has a `loc : SourcePos.t` and a
`ctx : Context.t`. Build a spatial index:

```
(* lib/hoverIndex.mli *)
type t
val of_file_result : LspCompile.file_result -> t
val lookup : t -> line:int -> col:int ->
  (SourcePos.t * Context.t * Sort.sort * Effect.t) option
  (* smallest enclosing typed node at the cursor *)
```

Under the hood `t` is a list of per-decl tries keyed by
`(start_line, start_col, end_line, end_col)`. Lookup walks the
trie for the decl whose span covers the cursor, then descends
into children until the smallest covering node is found.

#### 3.2.4 Async SMT

Current `SolverInvoke.run_z3` blocks. For LSP we need the server
loop to continue answering requests while Z3 churns.

**Design:**

```
(* lib/smtAsync.mli *)
type run_id = int
type event =
  | Query_result of { run : run_id; pos : SourcePos.t;
                      answer : SolverInvoke.answer }
  | Run_finished of run_id
  | Run_failed of run_id * string

val start :
  exe:string -> smt_path:string ->
  queries:(SourcePos.t list) ->
  on_event:(event -> unit) ->
  (run_id, string) result

val cancel : run_id -> unit
```

`start` spawns `z3 smt_path`, spawns a reader OCaml thread, and
returns the `run_id`. The reader thread parses `(check-sat)`
answers from stdout line-by-line (SMT-LIB output is
line-oriented: one top-level s-expression per verdict), calls
`on_event` on the main thread's wakeup pipe for each answer, and
emits `Run_finished` at EOF.

`cancel` sends `SIGTERM` to the subprocess and discards any
events whose `run_id` matches.

**Threading model.** OCaml threads (stdlib `threads.posix`) are
fine here: the reader thread performs blocking I/O and sends a
single byte down a `Unix.pipe` to wake the main loop. The main
loop uses `Unix.select` on `[stdin_fd; pipe_read_fd]`. All shared
mutable state (current `run_id`, pending queries) lives in a
single `Mutex.t`-guarded record. Zero CLAUDE.md violations
because this module is the *sole* place mutation is permitted,
behind a narrow interface.

**Concurrency invariants** (stated explicitly so QCheck tests
can encode them):

- At most one active run per file at any time.
- Each `Query_result` event lands in the event stream before
  `Run_finished` for the same run.
- `cancel r` guarantees no further events tagged `r` are
  delivered.

These are analogous to the "typed pipe" invariants already in
`solverInvoke.ml` and should QCheck well.

#### 3.2.5 Refactor of existing SMT code

Factor out answer-parsing from `solverInvoke.ml`:

```
(* lib/solverOutput.mli *)
val classify_line : string -> SolverInvoke.answer option
  (* returns Some if the line parses to one verdict *)
val classify_sexp : SmtSexp.sexp -> SolverInvoke.answer
```

Both the existing synchronous `run_z3` and the new `smtAsync`
call into `solverOutput`. Unchanged at the API boundary.

### 3.3 LSP feature mapping

| LSP request/notification       | nanoCN implementation                        |
|--------------------------------|----------------------------------------------|
| `textDocument/didOpen`         | Register in `Text_document`; compile_file.   |
| `textDocument/didChange`       | `Text_document.apply_content_changes`; debounce then recompile. |
| `textDocument/didSave`         | Fire SMT run for refined files.              |
| `publishDiagnostics`           | Derived from `LspCompile.decl_result.diagnostics` + SMT answers. |
| `textDocument/hover`           | `HoverIndex.lookup` → formatted ctx/sort/eff. |
| `textDocument/definition`      | Lookup via `Sig` / `ctx`; return Var's binding site. |
| `textDocument/documentSymbol`  | List `Sig` entries with loc. |
| `textDocument/references`      | (Phase 4) scan typed tree for Var occurrences. |
| `nanocn/context`               | Custom request; returns the full context object for "context at point". |
| `$/progress`                   | Wrap long SMT runs so Eglot can show a spinner. |

### 3.4 Debouncing

Recompile is triggered 150ms after the last `didChange`. SMT run
is triggered only on `didSave` (never on `didChange`). These
constants are exposed as LSP `initializationOptions`.

### 3.5 New OCaml module map

```
lib/splitter.ml{i}        depth-tracking decl splitter
lib/lspCompile.ml{i}      per-decl parse + typecheck driver
lib/hoverIndex.ml{i}      spatial trie over typed tree
lib/solverOutput.ml{i}    factored answer parser
lib/smtAsync.ml{i}        background Z3 driver
lib/lspProtocol.ml{i}     nanoCN ↔ LSP translations (pure)
bin/nanocn_lsp.ml         main loop
bin/dune                  adds `nanocn-lsp` executable
```

A new `RCheck.check_rdecl` binding is added but the existing
module otherwise unchanged.

### 3.6 Testing

- **Unit / QCheck** per new module, per CLAUDE.md rule 2.
  `Splitter`: round-trip (split + concat = original) on
  generated programs. `HoverIndex`: every position in a decl
  resolves to a node whose loc contains that position.
  `SmtAsync`: mock Z3 via a trivial echo process; assert event
  ordering invariants.
- **Integration**: a `test/lsp/` directory with scripted LSP
  sessions (jsonrpc messages) replayed against the executable,
  golden-compared.

## 4. Piece C — Emacs mode

### 4.1 Files

```
scripts/editor/emacs/
├── nanocn-ts-mode.el     # tree-sitter major mode
├── nanocn-lsp.el         # Eglot registration
├── nanocn-context.el     # "context at point" command + buffer
└── README.md
```

One `nanocn-ts-mode` handles both `.cn` and `.rcn`. Users who
want visually distinct variants can layer a simple minor mode on
top; not in scope here.

### 4.2 Major-mode skeleton (the standard Emacs-29 shape)

```elisp
(define-derived-mode nanocn-ts-mode prog-mode "nanoCN"
  "Major mode for editing nanoCN .cn/.rcn files."
  (when (treesit-ready-p 'nanocn)
    (treesit-parser-create 'nanocn)
    (setq-local treesit-font-lock-settings
                (apply #'treesit-font-lock-rules nanocn-ts-font-lock-rules))
    (setq-local treesit-font-lock-feature-list
                '((comment definition)
                  (keyword string type)
                  (constant number function builtin)
                  (operator delimiter bracket)))
    (setq-local treesit-simple-indent-rules
                nanocn-ts-indent-rules)
    (setq-local treesit-defun-type-regexp
                (rx bos (or "fun_decl" "rfun_decl"
                            "sort_decl" "type_decl") eos))
    (setq-local treesit-defun-name-function
                #'nanocn-ts-defun-name)
    (setq-local treesit-simple-imenu-settings
                nanocn-ts-imenu-settings)
    (treesit-major-mode-setup)))

;;;###autoload
(add-to-list 'auto-mode-alist '("\\.cn\\'" . nanocn-ts-mode))
;;;###autoload
(add-to-list 'auto-mode-alist '("\\.rcn\\'" . nanocn-ts-mode))
```

### 4.3 Font-lock feature levels

Match the Emacs-29 convention:

- Level 1 (`comment definition`): always on.
- Level 2 (`keyword string type`): default-on.
- Level 3 (`constant number function builtin`): default-on with
  decoration level 3.
- Level 4 (`operator delimiter bracket`): off by default,
  available to users who want everything lit up.

### 4.4 Indentation rules (starting set)

```elisp
(defvar nanocn-ts-indent-rules
  `((nanocn
     ((node-is "}") parent-bol 0)
     ((node-is "end_of_branch_bar") parent-bol 0)
     ((parent-is "source_file") column-0 0)
     ((parent-is "fun_decl") parent-bol 2)
     ((parent-is "case_branches") parent-bol 2)
     ((parent-is "iter_body") parent-bol 2)
     ((parent-is ,(rx "_block")) parent-bol 2)
     (no-node parent-bol 0))))
```

Exact node types are determined by `grammar.js`; the list is
refined while writing the grammar. `treesit--indent-verbose`
guides iteration.

### 4.5 LSP via Eglot

```elisp
;;;###autoload
(with-eval-after-load 'eglot
  (add-to-list 'eglot-server-programs
               '(nanocn-ts-mode . ("nanocn-lsp" "--stdio"))))
```

Eglot gives us diagnostics (via Flymake), hover (via ElDoc),
xref, Imenu, rename, formatting, and progress notifications
automatically. No custom code needed for 90% of the surface.

### 4.6 Context at point

The "list current context at point" feature is slightly richer
than a hover tooltip, so we implement it twice:

- **Hover** — Eglot's default. The server's `textDocument/hover`
  returns a Markdown block with `ctx : Ctx` as an indented list
  plus `sort : Sort` and `eff : Eff`. Emacs displays it in ElDoc.
- **Dedicated buffer** — `M-x nanocn-show-context`
  (bound to `C-c C-t` in the mode map) sends a custom
  `nanocn/context` request, gets a structured reply, and renders
  it in a side buffer `*nanoCN context*` that updates on cursor
  movement (optional, opt-in via `nanocn-context-mode`).

### 4.7 Grammar installation

We add a recipe to `treesit-language-source-alist` so users can
install the grammar via `M-x treesit-install-language-grammar`:

```elisp
;;;###autoload
(with-eval-after-load 'treesit
  (add-to-list 'treesit-language-source-alist
               '(nanocn "https://github.com/USER/tree-sitter-nanocn"
                        "main")))
```

(The grammar will live either in the nanocn repo at
`scripts/editor/tree-sitter-nanocn/` — in which case the URL
points there — or in its own repository. Open question; see §8.)

## 5. Shared infrastructure & testing

### 5.1 CI

Three new dune aliases:

- `check-grammar` — runs `tree-sitter parse` on every example and
  fails on `ERROR` / `MISSING`. Requires `tree-sitter` on PATH;
  guarded by `(enabled_if (<> %{env:TREE_SITTER_SKIP=0} 1))`.
- `check-lsp-session` — runs the golden LSP-message sessions.
- `test-smt-async` — runs the SMT-async mock tests (no real Z3).

### 5.2 Documentation

- `doc/editor-support.md` — user-facing install instructions.
- `doc/instructions/grammar-change.md` — reminds authors editing
  `parser.mly` to update `tree-sitter-nanocn/grammar.js`.

### 5.3 Versioning

Tree-sitter grammars aren't semantically versioned, so we tag the
nanoCN grammar with nanoCN compiler versions and document the
pairing in `scripts/editor/tree-sitter-nanocn/README.md`.

## 6. Phasing

**Phase 1 — Tree-sitter + syntax-only Emacs mode.** Minimal compiler
impact. Users get highlighting, indentation, Imenu, sexp
navigation. Deliverables: `tree-sitter-nanocn/`, `nanocn-ts-mode.el`,
`check-grammar` CI alias.

**Phase 2 — Synchronous LSP.** Build `nanocn-lsp` with:

- `splitter` + `lspCompile` + `hoverIndex`
- `publishDiagnostics` for parse + type errors
- `textDocument/hover`, `documentSymbol`, `definition`
- SMT runs still synchronous — they complete in the didSave
  handler before the response returns. Fine for small refined
  files.

**Phase 3 — Async SMT.** Add `smtAsync`; rewire the SMT path so
diagnostics stream in after didSave returns. Add
`$/progress` and `nanocn/context`.

**Phase 4 — Polish.** References, rename, completion, semantic
tokens, inlay hints (e.g. inferred sorts).

The three phases can overlap in development but the dependencies
are Phase 1 independent, Phase 2 depends on nothing new in the
compiler except the per-decl refactor, Phase 3 depends on Phase 2.

## 7. Risks and how they're mitigated

| Risk                                    | Mitigation                                                   |
|-----------------------------------------|--------------------------------------------------------------|
| Grammar drifts from `parser.mly`        | `check-grammar` CI alias + `doc/instructions/grammar-change.md` + the new grammar-change workflow mirrors the existing "Ott first" workflow. |
| Per-decl stub heuristics confuse users  | Phase 2 ships without stubs; failed decls simply drop from `Sig`. Add stubs only if users ask for them. |
| Incremental-typechecking correctness    | Recompile *whole file* on every change. Incremental per-decl would be a Phase 5 optimization. The hover-index rebuild is O(file size), fine for realistic files. |
| SMT process leakage                     | `smtAsync.cancel` uses SIGTERM; integration test asserts `ps` shows no orphaned z3. Add `SIGKILL` fallback after 2s. |
| Emacs users without tree-sitter (<29)   | `nanocn-ts-mode` errors out with a clear message pointing at `treesit-install-language-grammar`. |
| `lsp` opam lib API changes              | Pin a specific version in `nanocn.opam`. This lib has been stable for years. |
| Hand-rolled `Unix.select` loop subtle bugs | Keep it tiny (~50 lines); property tests for the splitter + deterministic mock SMT tests cover most of the state machine. |

## 8. Open questions for the user

1. **Grammar repo location.** Should the tree-sitter grammar live
   in the main nanocn repo at `scripts/editor/tree-sitter-nanocn/`,
   or in its own `tree-sitter-nanocn` repository? The latter is
   conventional (it's what every language in the tree-sitter
   org does) and lets `treesit-install-language-grammar` use a
   simple GitHub URL recipe. The former keeps everything in one
   place during early iteration. **Recommendation: start in the
   main repo; split out later if needed.**

Answer: Keep it in the main repo. 


2. **Shared-state module boundaries for `smtAsync`.** CLAUDE.md's
   module-by-module mutability allowance: are `smtAsync.ml` (for
   the thread-safe run state) and `solverInvoke.ml` (existing)
   acceptable impure modules? **Recommendation: yes — they are
   thin IO shells over the pure SMT encoder.**

Answer: yes

3. **Context-at-point display.** Ship hover + a
   `*nanoCN context*` side buffer (auto-updating as the cursor
   moves), or just hover? **Recommendation: ship both;
   side-buffer is opt-in.**

Answer: okay for now, but note that whole context may be too large (eg, entire spec function definitions 
	    can show up in types). The context display in a side buffer is more likely to tolerate big
	    terms. If it does get too large, then displaying either the inferred type of a term or the goal 
		type at point might be better. 

4. **Phasing.** Is the phasing above acceptable, or is there a
   reason to prioritize SMT-async earlier (e.g. because refined
   examples are the usual workload)?

Answer: yes. 

Additionally: 

5. **Parsing** Change the main CN parser to use the same definition-by-definition split, so that multiple
   parser errors can be reported in a single run of the compiler. The incremental style of the Menhir parser 
   should make it easy to advance to the next synchronization token (fun/sort/etc). 

## 9. Post-plan review — defects considered

The following defects were considered during plan authoring and
resolved in the text above:

1. **Unified grammar vs two grammars.** Chose unified; resolved
   in §2.2.
2. **Hyphenated keywords** (`open-ret`, etc.). Use explicit
   `token(...)` rules; resolved in §2.3.
3. **Menhir's weak error recovery.** Hand-rolled depth-tracking
   splitter; resolved in §3.2.1. Considered using tree-sitter
   on the compiler side — rejected as premature.
4. **Refined per-decl checking doesn't exist.** Extract
   `check_rdecl` from `check_rprog`; resolved in §3.2.1.
5. **Async SMT cancellation.** `SIGTERM` + tagged run IDs;
   resolved in §3.2.4.
6. **Diagnostic correctness under race**. Each run gets a
   monotonic id; only latest-run diagnostics are published;
   resolved in §3.2.4.
7. **Hover during running SMT.** Hover uses the already-computed
   typed tree, not the SMT state. No interaction.
8. **Tree-sitter `.so` discovery in Emacs.** Ship a
   `treesit-language-source-alist` recipe; resolved in §4.7.
